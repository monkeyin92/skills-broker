# Architecture

**Analysis Date:** 2026-04-22

## Pattern Overview

**Overall:** Thin-host + shared-home + centralized broker runtime.

**Key Characteristics:**
- Hosts only decide the coarse boundary (`broker_first`, `handle_normally`, `clarify_before_broker`) via generated shell skill content from `src/hosts/skill-markdown.ts`; final capability selection stays inside `src/broker/run.ts`.
- Request normalization, candidate discovery, ranking, prepare, workflow state, and handoff are separated into focused modules under `src/core/`, `src/sources/`, and `src/broker/`.
- Runtime state is file-backed and relocatable: cache, workflow sessions, acquisition memory, verified downstream manifests, and routing traces live under the shared broker home managed by `src/shared-home/install.ts`, `src/shared-home/update.ts`, and `src/shared-home/doctor.ts`.

## System Boundary

**Inside the broker system:**
- Host-facing request/output contracts in `src/core/envelope.ts`, `src/core/request.ts`, `src/core/workflow.ts`, and `src/broker/result.ts`
- Capability discovery, ranking, workflow execution, and handoff in `src/broker/*.ts`
- Host shell installation and adapter execution in `src/hosts/claude-code/*` and `src/hosts/codex/*`
- Shared runtime installation, repair, and diagnostics in `src/shared-home/*.ts`
- Seed catalogs and maintained-family contract in `config/host-skills.seed.json`, `config/mcp-registry.seed.json`, and `config/maintained-broker-first-families.json`

**Outside the broker system:**
- The host model loop in Claude Code or Codex that decides whether to invoke the broker at all, based on the generated `SKILL.md`
- Execution of downstream skills after handoff; the broker returns a `HandoffEnvelope` but does not execute the selected skill itself
- External MCP registry content consumed by `src/sources/mcp-registry.ts`
- Repo status proof rails consumed by `src/shared-home/status.ts` from `STATUS.md`

## Layers

**Host Shell Layer:**
- Purpose: expose a host-native shell surface and runner that forwards coarse broker requests
- Location: `src/hosts/claude-code/adapter.ts`, `src/hosts/claude-code/install.ts`, `src/hosts/codex/adapter.ts`, `src/hosts/codex/install.ts`, `src/hosts/skill-markdown.ts`
- Contains: install scripts, thin-host runner scripts, adapter wrappers around `bin/run-broker`, generated host instructions
- Depends on: `src/broker/run.ts`, `src/cli.ts`, `src/core/maintained-broker-first.ts`, `src/shared-home/ownership.ts`
- Used by: installed host shells under `~/.claude/skills/skills-broker` and `~/.agents/skills/skills-broker`, plus smoke tests in `tests/e2e/shared-home-smoke.test.ts` and `tests/e2e/workflow-host-smoke.test.ts`

**Core Contract Layer:**
- Purpose: define the stable request, capability, workflow, and host support model
- Location: `src/core/`
- Contains: `BrokerEnvelope`, `CapabilityCard`, `WorkflowRecipe`, `BrokerHost` support metadata, cache policy, maintained-family contract validation
- Depends on: Node built-ins only
- Used by: every other source directory

**Discovery and Routing Layer:**
- Purpose: turn a normalized request into a selected capability or workflow
- Location: `src/broker/` and `src/sources/`
- Contains: normalization support, candidate merging, package availability probing, ranking, semantic tie-breaks, prepare, handoff assembly, workflow runtime, source adapters
- Depends on: `src/core/*`, `config/*.json`, shared-home state files, local skill directories, recorded MCP registry payloads
- Used by: `src/cli.ts` and host adapters

**Shared Home Lifecycle Layer:**
- Purpose: install and maintain the shared broker runtime plus audited host shells
- Location: `src/shared-home/`
- Contains: installer layout resolution, update/remove flows, doctor diagnostics, peer-surface migration, ownership manifests, routing-trace rollups, status board checks
- Depends on: `src/hosts/*`, `src/broker/*`, git and filesystem state, repo-local `STATUS.md`
- Used by: lifecycle CLI in `src/bin/skills-broker.ts`

**Configuration Layer:**
- Purpose: seed the broker with known packages, skills, workflows, and maintained routing proofs
- Location: `config/`
- Contains: host skill catalog, MCP registry seed, maintained broker-first families
- Depends on: validation in `src/sources/host-skill-catalog.ts`, `src/sources/mcp-registry.ts`, and `src/core/maintained-broker-first.ts`
- Used by: runtime discovery, host shell generation, doctor proof rails

## Data Flow

**Direct Capability Handoff:**

1. The installed host shell generated by `src/hosts/skill-markdown.ts` decides `broker_first` and sends a `BrokerEnvelope` to the local runner written by `src/hosts/claude-code/install.ts` or `src/hosts/codex/install.ts`.
2. `src/cli.ts` parses the JSON envelope, resolves `currentHost`, and forwards it to `runBroker` in `src/broker/run.ts`.
3. `src/core/request.ts` and `src/broker/query-compiler.ts` normalize raw text or a provided `capabilityQuery` into a query-native request; `src/broker/resolved-request.ts` derives a canonical query identity for cache and reuse.
4. `src/broker/run.ts` loads candidates from `src/sources/host-skill-catalog.ts`, `src/sources/mcp-registry.ts`, `src/broker/acquisition-memory.ts`, and `src/broker/downstream-manifest-source.ts`, then merges them with precedence rules from `src/broker/discover.ts`.
5. `src/broker/package-availability.ts` probes local/broker-managed skill roots, `src/broker/rank.ts` scores candidates, `src/broker/prepare.ts` validates the winner, and `src/broker/handoff.ts` returns `HANDOFF_READY` with explicit package, leaf, and implementation selection.

**Broker-Owned Workflow:**

1. Workflow candidates are declared in `config/host-skills.seed.json` and materialized as `WorkflowRecipe` objects by `src/sources/host-skill-catalog.ts`.
2. If `src/broker/run.ts` ranks a `broker_workflow` winner, it persists a run in `src/broker/workflow-session-store.ts` and delegates stage execution to `src/broker/workflow-runtime.ts`.
3. `src/broker/workflow-runtime.ts` returns `WORKFLOW_STAGE_READY`, `WORKFLOW_BLOCKED`, `WORKFLOW_COMPLETED`, or `WORKFLOW_FAILED`, depending on stage prerequisites, install state, artifact declarations, and optimistic-lock session updates.
4. The host resumes a workflow by sending `workflowResume` in a new `BrokerEnvelope`; the runtime verifies `runId`, `stageId`, submitted artifacts, and revision freshness before advancing.

**Cross-Host Reuse and Replay:**

1. Successful routing writes a short-lived winner cache through `src/core/cache/store.ts`, plus advisory memory through `src/broker/acquisition-memory.ts` and `src/broker/downstream-manifest-source.ts`.
2. The next request on either host can replay these advisory sources before catalog or registry discovery, because `src/broker/run.ts` loads both acquisition memory and verified downstream manifests ahead of seed catalogs.
3. `src/broker/local-skill-handoff.ts` can resolve a local skill from the current host or another managed host when `brokerHomeDirectory` is present, allowing cross-host reuse without re-discovery.

**Lifecycle Install / Repair / Audit:**

1. `src/bin/skills-broker.ts` parses `update`, `doctor`, and `remove`, resolves target paths through `src/shared-home/paths.ts`, and dispatches to the shared-home layer.
2. `src/shared-home/install.ts` creates the shared runtime under `~/.skills-broker`, copies `dist/` and `config/`, and writes the shared `bin/run-broker`.
3. `src/shared-home/update.ts` installs host shells, records ownership with `src/shared-home/ownership.ts`, migrates competing peer skills with `src/shared-home/host-surface.ts`, and repairs downstream runtime issues when possible.
4. `src/shared-home/doctor.ts` audits routing traces, acquisition memory, verified downstream manifests, broker-first proof gates, and repo status proofs, then reports adoption health for both hosts.

**State Management:**
- The winner cache is a single JSON file managed by `src/core/cache/store.ts`; without a broker home it falls back to a temp-file path from `src/broker/run.ts`.
- Workflow runs are stored as one JSON file per run under `workflow-sessions.json.d/` by `src/broker/workflow-session-store.ts`, with lock files and revision checks.
- Long-lived proof rails live under the shared home `state/` directory: acquisition memory from `src/broker/acquisition-memory.ts`, routing traces from `src/broker/trace-store.ts`, broker-first gate artifacts from `src/shared-home/broker-first-gate.ts`, and doctor-readable status snapshots from `src/shared-home/doctor.ts`.

## Key Abstractions

**BrokerEnvelope:**
- Purpose: host-to-broker request contract
- Examples: `src/core/envelope.ts`, `src/cli.ts`, `tests/cli/cli-contract.test.ts`
- Pattern: one envelope can carry raw request text, a structured `capabilityQuery`, a `workflowResume`, and downstream `executionFailures`, but `capabilityQuery` and `workflowResume` are mutually exclusive

**CapabilityCard:**
- Purpose: normalize skills, MCP servers, and workflow winners into a single ranking model
- Examples: `src/core/capability-card.ts`, `src/broker/rank.ts`, `src/sources/host-skill-catalog.ts`
- Pattern: every candidate exposes package identity, leaf identity, query metadata, implementation type, host support, prepare flags, and ranking hints

**ResolvedBrokerRequest:**
- Purpose: canonicalize request identity and compatibility intent for cache and reuse
- Examples: `src/broker/resolved-request.ts`, `src/broker/run.ts`
- Pattern: query-native identity is derived from artifacts, job families, targets, and preferred capability, then reused for cache and acquisition memory lookups

**WorkflowRecipe and WorkflowSession:**
- Purpose: represent broker-owned multi-step execution with resumable state
- Examples: `src/core/workflow.ts`, `src/broker/workflow-runtime.ts`, `src/broker/workflow-session-store.ts`
- Pattern: recipes are catalog-defined, sessions are stateful JSON records with `runId`, `activeStageId`, produced artifacts, and optimistic-lock revisions

**HandoffEnvelope:**
- Purpose: broker-to-host execution contract after a winner is selected
- Examples: `src/broker/handoff.ts`, `src/broker/local-skill-handoff.ts`, `tests/broker/handoff.test.ts`
- Pattern: carries `brokerDone`, explicit chosen package/leaf/implementation, optional local skill path, current host context, and the normalized request that justified the handoff

**MaintainedBrokerFirstContract:**
- Purpose: keep host coarse-boundary examples aligned with expected winners and proof rails
- Examples: `config/maintained-broker-first-families.json`, `src/core/maintained-broker-first.ts`, `src/shared-home/broker-first-gate.ts`
- Pattern: validated JSON contract feeds both generated host-shell instructions and doctor-time gate validation

## Host and Broker Collaboration

**Host-side coarse decision contract:**
- `src/hosts/skill-markdown.ts` generates the installed `SKILL.md` that tells the host to pick only `broker_first`, `handle_normally`, or `clarify_before_broker`.
- Claude Code installs that shell in auto mode through `src/hosts/claude-code/install.ts`; Codex installs it in explicit mode through `src/hosts/codex/install.ts`.
- The host must not choose a package, workflow, skill, or MCP winner on its own once it enters `broker_first`.

**Runner invocation contract:**
- Host runners created by `src/shared-home/install.ts`, `src/hosts/claude-code/install.ts`, and `src/hosts/codex/install.ts` set `BROKER_CURRENT_HOST` and execute the shared `bin/run-broker`.
- `src/hosts/claude-code/adapter.ts` and `src/hosts/codex/adapter.ts` mirror that contract in tests by `execFile`-ing the installed runner and parsing JSON stdout.

**Broker response contract:**
- `HANDOFF_READY` means the host should execute the selected downstream path, optionally reading `handoff.localSkill.skillFilePath`.
- `WORKFLOW_STAGE_READY` means the host should execute the returned stage or downstream skill and later resume with `workflowResume`.
- `INSTALL_REQUIRED`, `NO_CANDIDATE`, `UNSUPPORTED_REQUEST`, `AMBIGUOUS_REQUEST`, and `PREPARE_FAILED` map to explicit `hostAction` values in `src/broker/result.ts` and `src/broker/trace.ts`.
- If downstream execution fails, the host is expected to rerun the same request with `executionFailures`, allowing `src/broker/run.ts` to exclude the failed candidate and retry routing.

## Entry Points

**Lifecycle CLI:**
- Location: `src/bin/skills-broker.ts`
- Triggers: `skills-broker update`, `skills-broker doctor`, `skills-broker remove`
- Responsibilities: resolve target paths, call shared-home lifecycle methods, enforce strict doctor gating, and format human/JSON output

**Broker CLI:**
- Location: `src/cli.ts`
- Triggers: local runner scripts under `bin/run-broker`
- Responsibilities: parse broker-envelope JSON, resolve host context, call `runBroker`, and optionally include routing traces

**Host Adapters:**
- Location: `src/hosts/claude-code/adapter.ts`, `src/hosts/codex/adapter.ts`
- Triggers: smoke/integration tests and any programmatic embedding of installed shells
- Responsibilities: verify the installed shell layout, invoke the runner, and deserialize broker output

**Config-backed Discovery:**
- Location: `src/sources/host-skill-catalog.ts`, `src/sources/mcp-registry.ts`
- Triggers: every broker discovery pass
- Responsibilities: validate catalog shape, materialize workflow recipes, and normalize MCP servers into broker candidates

## Error Handling

**Strategy:** Convert normalization, discovery, prepare, and workflow failures into structured broker outcomes instead of leaking raw exceptions to the host.

**Patterns:**
- `src/core/request.ts` throws typed `UnsupportedBrokerRequestError` and `AmbiguousBrokerRequestError`; `src/broker/run.ts` translates them into `UNSUPPORTED_REQUEST` or `AMBIGUOUS_REQUEST`.
- `src/broker/prepare.ts` throws `PrepareCandidateError` with a machine-readable `reasonCode`; `src/broker/run.ts` converts it into `PREPARE_FAILED` plus trace metadata.
- `src/broker/workflow-session-store.ts` uses per-run lock files and revision checks; stale resume attempts become `WORKFLOW_FAILED` with `STALE_STAGE`.
- Advisory persistence in `src/broker/run.ts` and `src/broker/workflow-runtime.ts` is best-effort; failure to write traces or reuse memory does not abort routing.
- Shared-home repair records manual recovery instead of silently hiding filesystem conflicts, using `src/shared-home/host-surface.ts` and `src/shared-home/peer-surface-audit.ts`.

## Cross-Cutting Concerns

**Logging:** Routing traces are created by `src/broker/trace.ts`, persisted as JSONL by `src/broker/trace-store.ts`, and summarized by `src/shared-home/doctor.ts`.

**Validation:** Request validation lives in `src/core/envelope.ts` and `src/core/request.ts`; catalog validation lives in `src/sources/host-skill-catalog.ts`; maintained-family validation lives in `src/core/maintained-broker-first.ts`.

**Authentication:** The broker itself has no first-party auth layer. It only carries prepare flags (`authRequired`, `installRequired`) inside `CapabilityCard`; actual authentication and credential handling belong to downstream skills or MCP servers outside this repo.
